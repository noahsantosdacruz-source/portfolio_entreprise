Journal de mise en œuvre

Du tunnel VPN à l'écran allumé

Reconstitution des étapes réelles de la session de déploiement, dans l'ordre où les problèmes se sont posés et ont été résolus.

1

Authentification OpenVPN résolu

La ligne auth-user-pass de xibo.conf ne pointait pas vers le bon fichier d'identifiants. Correction par sed, puis vérification et redémarrage du service.

# pointer explicitement vers le fichier de login
sudo sed -i 's|^auth-user-pass$|auth-user-pass /etc/openvpn/client/xibo-login.txt|' /etc/openvpn/client/xibo.conf
sudo systemctl restart openvpn-client@xibo
2

Adresse du CMS résolu

Deux adresses ont circulé pendant les tests : 192.168.1.62 (réseau local) et 192.168.200.38, l'adresse réellement joignable une fois le tunnel VPN monté. Seule la seconde fonctionne en dehors du réseau local.

3

Chromium sous Wayland résolu

Lancé sans les bonnes variables d'environnement, Chromium ne trouvait pas de session graphique. Il fallait exporter XDG_RUNTIME_DIR et WAYLAND_DISPLAY avant l'appel.

export XDG_RUNTIME_DIR=/run/user/1000
export WAYLAND_DISPLAY=wayland-0
chromium --ozone-platform=wayland \
  --user-data-dir=/home/pi/.config/chromium-neuf \
  --start-fullscreen http://192.168.200.38
4

Transfert des fichiers vers le Pi résolu

Les commandes scp échouaient car elles étaient lancées directement sur le Raspberry Pi plutôt que depuis la machine où résidaient les fichiers. Les scripts ont finalement été créés sur place avec nano.

5

Démarrage automatique résolu

Un service systemd dédié (xibo-player-auto.service) lance le script après que le réseau et le VPN soient montés, avec redémarrage automatique en cas d'arrêt. Testé par redémarrage complet du Raspberry Pi.

Configuration finale

Les deux fichiers qui font tout le travail

Un script shell pour lancer Chromium proprement, et une unité systemd pour le surveiller.

/home/pi/lancer-xibo.sh
#!/bin/bash
# Attendre que le réseau soit bien monté par le VPN
sleep 10

# Lancer l'affichage en plein écran
export XDG_RUNTIME_DIR=/run/user/1000
export WAYLAND_DISPLAY=wayland-0

chromium --ozone-platform=wayland \
  --user-data-dir=/home/pi/.config/chromium-neuf \
  --start-fullscreen http://192.168.200.38
/etc/systemd/system/xibo-player-auto.service
[Unit]
Description=Lancement auto Xibo Client
After=network-online.target openvpn-client@xibo.service

[Service]
User=pi
ExecStart=/home/pi/lancer-xibo.sh
Restart=always

[Install]
WantedBy=graphical.target
Activation
chmod +x /home/pi/lancer-xibo.sh
sudo systemctl daemon-reload
sudo systemctl enable xibo-player-auto.service
sudo systemctl start xibo-player-auto.service
Référence

Dépannage courant

Symptômes rencontrés pendant le déploiement et ce qu'ils signifiaient réellement.

SymptômeCauseAction
read UDPv4 [EHOSTUNREAC] Le serveur VPN n'est pas joignable depuis le réseau actuel (routage ou pare-feu). Vérifier la connectivité réseau du Pi avant d'incriminer la configuration OpenVPN.
Fontconfig error: Cannot load default config file Avertissement sans conséquence sur l'affichage — Chromium ne trouve pas de config de polices système. Ignorer, ou installer fontconfig si des glyphes manquent réellement à l'écran.
Registration response error: DEPRECATED_ENDPOINT Échec du service de notifications push de Chromium (GCM), inutilisé en kiosque. Ignorer sans risque.
scp: stat local: No such file or directory La commande scp était exécutée depuis le Raspberry Pi et non depuis la machine source. Lancer scp depuis la machine qui possède les fichiers, ou créer le fichier sur place.
Chromium ne s'affiche pas malgré une commande correcte XDG_RUNTIME_DIR et WAYLAND_DISPLAY non définis dans le contexte de lancement (ex. service systemd). Les exporter explicitement dans le script de démarrage, pas seulement dans le shell interactif.
